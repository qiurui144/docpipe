# docpipe v1.1 — API 固化 + 端到端可用 设计规范

**日期**: 2026-06-24 · **状态**: 已批准（设计阶段）· **作者**: qiurui144

---

## 1. 目标定位

当前 docpipe HTTP API（`/v1/{parse,chunk,embed,search,annotate,health}`）只是一组**无状态变换 + 一个搜不到东西的 search**：facade 有 `ingest()`（parse→chunk→embed→store）但**未暴露 HTTP**，也没有任何写入向量库 / 文档管理的端点。因此"上传文档 → 检索"这条 RAG 核心链**经 HTTP 走不通**。

v1.1 的目标：把已有 `ingest()` 接出 HTTP + 加文档生命周期管理 + 极简异步任务 + 固化 API 契约，让 API **端到端可用且稳定易调用**。与产品定位对齐：docpipe 是通用文档管线 SDK，"摄入并检索文档"是其最核心承诺。

## 2. 范围边界

### 本版本（v1.1）做
- `POST /v1/ingest`：multipart 文件一步 parse→chunk→embed→store；同步返回结果，或 `async` 返回 job_id。
- 文档登记表 + 管理：`GET /v1/documents`、`GET /v1/documents/{doc_id}`、`DELETE /v1/documents/{doc_id}`。
- 极简进程内异步 job：`GET /v1/jobs/{job_id}`。
- 契约固化：请求体大小上限、错误码补全、`/v1` 稳定性声明、openapi 对齐 + 覆盖测试。
- Python / TS 客户端：新方法 + typed 错误。

### 不做（v1.1）—— 留后续
- EPUB 解析、Weaviate 后端、**持久化 job**（重启不丢）、模型自动下载、表格→CSV/JSON。
- 鉴权 / 多租户（API 层暂无 auth，由部署方网关负责）。

## 3. 架构数据流

```
POST /v1/ingest (multipart file + config)
   │  config.async == false（默认，中小文件）
   ├──────────────► facade.parse → facade.ingest → store(documents + vec)
   │                  └─► 200 {doc_id, collection, chunk_count, chunk_ids, backend, ocr_used}
   │  config.async == true（大文件 / 长 OCR）
   └─► JobQueue.submit(job) ──► 202 {job_id, status:"queued"}
            │  tokio task（Semaphore(MAX_OCR_CONCURRENCY) 限并发）
            └─► parse+ingest，更新 job 状态
                 GET /v1/jobs/{job_id} ─► {status, result?, error?}

GET    /v1/documents?collection=  ─► store.list_documents  ─► [{doc_id, filename, format, page_count, chunk_count, created_at}]
GET    /v1/documents/{doc_id}     ─► store.get_document
DELETE /v1/documents/{doc_id}     ─► store.delete_document（删 documents 行 + vec_chunks + chunk_meta）
```

**store schema 增量**（SqliteVecStore）：新增 `documents` 表
```sql
CREATE TABLE IF NOT EXISTS documents (
  doc_id TEXT NOT NULL, collection TEXT NOT NULL,
  filename TEXT, format TEXT, page_count INTEGER, chunk_count INTEGER,
  created_at TEXT NOT NULL,           -- RFC3339（由调用方传入，core 不取系统时钟）
  PRIMARY KEY (doc_id, collection)
);
```

**JobQueue**（进程内）：`Arc<Mutex<HashMap<String, Job>>>` + `tokio::Semaphore`。`Job { id, status: Queued|Running|Done|Failed, created_at, result: Option<IngestResult>, error: Option<String> }`。重启即失（文档明示）。

## 5. API 契约

### POST /v1/ingest
```
Request: multipart/form-data  file=<binary>  config=<JSON ParseConfig + {async?:bool, collection?:string}>
200 (sync): {"doc_id","collection","chunk_count":N,"chunk_ids":[...],"backend","ocr_used":bool}
202 (async): {"job_id","status":"queued"}
413 file-too-large（超 MAX_UPLOAD_BYTES，默认 500MB）/ 422 parse-empty-result / 502 embedding-failed
```
### GET /v1/documents?collection=default
```
200: {"documents":[{"doc_id","collection","filename","format","page_count","chunk_count","created_at"}]}
```
### GET /v1/documents/{doc_id}?collection=default
```
200: DocumentInfo ; 404 document-not-found
```
### DELETE /v1/documents/{doc_id}?collection=default
```
200: {"deleted":true,"doc_id"} ; 404 document-not-found
```
### GET /v1/jobs/{job_id}
```
200: {"job_id","status":"queued|running|done|failed","created_at","result?":IngestResult,"error?":string}
404 job-not-found
```

### Rust facade（新增）
`async fn ingest_file(&self, bytes, filename, config) -> Result<IngestResult>`（parse+ingest+register 一步）、`async fn list_documents(collection)`、`async fn get_document(doc_id, collection)`、`async fn delete_document(doc_id, collection)`。

## 6. 扩展点 / 兼容
- `/v1` **只增不改**：新增端点 + 既有响应只加字段；破坏性变更走 `/v1/v2`（文档声明）。
- 持久化 job / Weaviate 文档登记是后续 trait 扩展点（JobStore trait、VectorStore 已有 register_document 钩子）。
- 旧 6 端点行为不变（向后兼容）。

## 7. 错误与边界 case
| 码 | HTTP | 触发 |
|---|---|---|
| `file-too-large` | 413 | 上传超 MAX_UPLOAD_BYTES |
| `document-not-found` | 404 | get/delete 不存在的 doc_id |
| `job-not-found` | 404 | 查询不存在的 job_id |
| `parse-empty-result` | 422 | ingest 解析为空 |
| `embedding-failed` | 502 | ingest 向量化失败 |

边界：空文件→422；async job 失败→job.status=failed + error（不 panic）；重复 ingest 同 doc_id→幂等（delete 旧 + 重写，documents 行 upsert）；删不存在 doc→404；超并发 job→Semaphore 排队（不拒绝）。

## 8. 成本契约
零额外 API 成本（全本地）。异步 job 仅占内存（job 表 + tokio task），无额外容器。MAX_OCR_CONCURRENCY 限同时 OCR，保对话实时性（后台 job 不抢占）。

## 9. 测试矩阵
| 类 | 用例 |
|---|---|
| happy | ingest 文本层 PDF（同步）→ documents 出现 → search 命中 → delete → documents 空 |
| happy | ingest async → job queued→running→done，result 含 doc_id |
| edge | 空文件 ingest → 422；删不存在 doc → 404；查不存在 job → 404 |
| edge | 重复 ingest 同源 → documents 一行、chunk 不翻倍（幂等） |
| error | 超大小限制 → 413；embedder 不可达 → 502 + job failed |
| contract | **openapi 覆盖测试**：所有 router 路由都在 openapi.yaml 有定义（防漂移） |
| client | python/ts：ingest_file、wait_for_job、list/get/delete_document、get_job、typed 错误 |
| e2e（真机） | Windows Intel：起服 → ingest 扫描中文 PDF（OCR）→ documents → search 命中 → async job → delete |

## 10. 向后兼容
旧 6 端点签名/响应不变。store 加 `documents` 表用 `CREATE IF NOT EXISTS`，旧 db 自动建表、旧数据（无 documents 行）list 时不显示但 search 仍可用（向后兼容降级）。客户端旧方法保留，新方法只加。

## 11. 风险登记
| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| 内存 job 重启丢失 | 高 | 低 | 文档明示 v1.1 ephemeral；持久化留后续 |
| 大文件同步 ingest 阻塞 | 中 | 中 | async 路径 + 大小上限 + 超时；文档建议大文件用 async |
| documents 表与 chunk 数据不一致（部分失败） | 中 | 中 | ingest 顺序：先 store 向量成功再 register_document；delete 同事务删三处 |
| openapi 再漂移 | 中 | 低 | 覆盖测试做 CI 门 |
| job task panic 拖垮 server | 低 | 中 | task 内 catch → job.status=failed，不向上 panic |
